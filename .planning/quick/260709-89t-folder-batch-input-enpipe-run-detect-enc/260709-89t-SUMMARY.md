# Quick Task 260709-89t: Folder Batch Input for enpipe run/detect/encode Summary

**`enpipe run|detect|encode <папка>` now batch-processes every video file in a directory (optionally recursive), skipping already-done files and continuing past per-file failures, while the single-file/nonexistent-path code path stays byte-identical to before.**

- **Tasks:** 3

## Accomplishments
- New leaf module `src/enpipe/shared/batch.py`: `VIDEO_EXTS` allowlist (12 common container suffixes), `iter_input_videos(path, recursive)` (single file passthrough / directory allowlist-scan minus self-outputs / sorted / empty-or-nonexistent -> `[]`), and `run_batch(videos, process_one, label, should_skip)` implementing collect-then-report — one file's `SystemExit` or `Exception` is logged and the batch continues, with a final `die()` (nonzero exit) if any files failed.
- `enpipe run <dir>`: `run_pipeline`'s detect->encode body was extracted into `_pipeline_one(video, scenes_path, args)` (byte-identical two-Namespace construction), then wrapped by a new directory branch that runs the output-collapse guards, discovers videos, and drives them through `run_batch` with `should_skip` keyed on `resolve_output_path(...).exists()`.
- `enpipe detect <dir>`: guards `-o` (die if set — `.scenes` must live next to each source file), then batches via `iter_input_videos`/`run_batch`, recursively calling `run_detect` per file with a fresh `Namespace(input=v, output=None, ...)`; skips files that already have a sibling `.scenes`.
- `enpipe encode <dir>`: which-preflight stays first (unchanged), then guards `scenes`/`-o`-as-file/`--workdir`/`--csv`, batches via `iter_input_videos`/`run_batch`, recursively calling `run_encode` per file with `scenes=<v>.scenes`; skips files with no sibling `.scenes` ("нет .scenes", not a failure) or an already-existing resolved output ("уже готов").
- `argparse`: `--recursive` (store_true) added to all three subparsers; `encode`'s positional `scenes` became `nargs="?"` defaulting to `None` — `run_encode`'s single-file branch now defaults `args.scenes` to `<video>.scenes` itself (previously this default only existed inside `run_pipeline`).
- All Russian die()-messages explain *why* the guard exists (output-path collision -> silent video loss), matching the codebase convention of justifying non-obvious engineering decisions inline.

## Decisions Made
- Extracted `_pipeline_one` from `run_pipeline` rather than inlining batch logic around the existing body, so the exact same Namespace-construction code serves both the single-file and per-video-in-batch call sites with zero duplication risk.
- `run_detect`/`run_encode` dispatch to themselves recursively for each file in a directory (via `Namespace(**{**vars(args), "input"/"video": v, ...})`) rather than factoring out a separate `_process_one` helper — keeps the single-file logic as the one and only implementation, with the directory branch as a thin wrapper, matching the plan's explicit interface spec.
- Guards for output-collapsing arguments are evaluated in the directory branch of both `run_pipeline` and `run_encode` independently (duplicated ~4 lines each) rather than factored into `shared/batch.py`, because the exact set of arguments differs slightly between `run` (`args.scenes`) and `encode` alone (`args.scenes`) — plan's interface spec called these out per-function rather than as a shared verb.
