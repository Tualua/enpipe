<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CI runner → GHCR | GITHUB_TOKEN-authenticated push of a built image to the org registry |
| GitHub-hosted action → workflow | Third-party `docker/*` action code runs in the job with `packages: write` |
| Build → GitHub API | BuildKit secret (token) crosses into the build to authenticate release downloads |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-hq2-01 | Tampering | third-party `docker/*` actions | mitigate | Pin every docker/* action by full 40-hex commit SHA (not a moving tag), with `# vX.Y.Z` comment — mirrors setup-uv pinning in ci.yml |
| T-hq2-02 | Elevation of Privilege | `packages: write` on a tag/dispatch-triggered job | mitigate | Trigger only on `push: tags: v*` and `workflow_dispatch` (no `pull_request` from forks); `permissions` scoped to `contents: read` + `packages: write` only |
| T-hq2-03 | Information Disclosure | GITHUB_TOKEN passed as BuildKit `secrets:` | mitigate | Use `--mount=type=secret` (never `ARG`/`ENV`), so the token is not baked into image layers or logs; token is the ephemeral job-scoped GITHUB_TOKEN, not a long-lived PAT |
| T-hq2-04 | Spoofing | image consumers pulling `:latest` | accept | Provenance/signing (cosign) out of scope for this quick task; `type=sha` tag gives a traceable immutable ref |
| T-hq2-SC | Tampering | npm/pip/cargo installs | accept | No package-manager installs added; Dockerfile edit is auth-only, no new deps (per constraints) |
</threat_model>
