# Quick Task 260709-711: Smooth Per-Frame Progress Bar Summary

**Aggregated tqdm bar in `detect_scenes_parallel(show_progress=True)` now advances per-frame via a `progress_cb` hook threaded through `QsvPipeStream.read()`, replacing the old `as_completed`/per-segment-completion jump.**

## Performance

- **Duration:** ~20 min
- **Completed:** 2026-07-09
- **Tasks:** 3
- **Files modified:** 3 (+1 new test file)

## Accomplishments
- `QsvPipeStream.__init__` gained an optional `progress_cb: Optional[Callable[[int], None]] = None` parameter; `read()` calls it exactly once per successfully-read frame (including `decode=False` reads used by segment-boundary probing), and never on a truncated/EOF read. `progress_cb=None` (default) is a strict no-op — zero behavior change for every existing caller.
- `detect_scenes_parallel`'s `show_progress=True` branch dropped the `as_completed`/`span_by_future` per-segment-completion update in favor of one shared `tqdm` bar (`cb = bar.update`) driven by per-frame callbacks from every segment thread's `QsvPipeStream.read()` concurrently (`tqdm.update` is thread-safe). `as_completed` import removed entirely (unused after the change).
- `_segment_worker` now accepts both the legacy 4-tuple (`show_progress=False`, `ex.map` path — byte-identical, unchanged) and a new 5-tuple with `progress_cb` as the optional trailing element.
- `results = [f.result() for f in futures]` still collects strictly in `futures[i]` submission order (not completion order) — the correctness-critical invariant for the offset/non-cut-boundary merge math in steps 3-4 is unchanged and unexercised by this task.
- New hardware-free unit test `tests/unit/detection/test_stream_progress.py` exercises `QsvPipeStream.read()`'s callback behavior directly (via `object.__new__` + fake `stdout`), without touching `probe_source`/`ffmpeg`/subprocess.
- `tests/unit/detection/test_show_progress.py` updated: `_fake_as_completed` mock removed (symbol no longer exists in `parallel.py`), `_fake_segment_worker` now simulates per-frame reads by calling `cb(1)` `_SEG_LEN` times when a 5th-element callback is present, preserving the `sum(updates) == _TOTAL_FRAMES` regression assertion under the new model.
- `tests/unit/detection/test_parallel_merge.py` was **not** modified and remains green — it only exercises the `.map`/4-tuple path, confirming the `show_progress=False` branch is byte-identical.

## Decisions Made
- `progress_cb` fires before the `decode`/`not decode` branch in `read()`, so `decode=False` calls (used internally by `_detect_relative`'s frame-skip fast path and by `find_boundary`'s segment probes) still count toward the aggregated total — matches the plan's explicit interface spec.
- Kept the strict `futures[i]`-order result collection unchanged; only the *bar-driving* mechanism changed, not the *result-collection* mechanism — these are two independent concerns in the original code that the plan (and this execution) deliberately kept separated.
- `_segment_worker`'s tuple-unpacking uses `args[0..3]` positional indexing plus `args[4] if len(args) > 4 else None` rather than a variadic `*rest` unpack, matching the plan's literal interface spec exactly.
