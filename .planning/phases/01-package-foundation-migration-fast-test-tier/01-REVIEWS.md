# Cross-AI Plan Review: Phase 1 (3 Waves)

## 1. Summary

All three plans are exceptionally well-structured for a packaging + mechanical-migration phase. They faithfully implement every CONTEXT.md decision (D-01 through D-15), maintain tight scope discipline, and provide clear line-level source maps, explicit parity verification, and a clean wave dependency chain. The plans avoid the most common refactoring pitfall — rewriting algorithms instead of moving them — and the `shared.proc` / `shared.logging` split is the right minimal seam. Minor gaps exist around the encode-parity script's reliance on real GPU hardware (contradicting the "fast test tier" premise slightly) and a thin spot in the detection parity script's handling of the `--no-qsv` fallback path.

## 2. Strengths

- **Zero-logic-change discipline enforced at every layer.** Line-level source→module maps, verbatim copy directives, and the `encode_scenes: ` prefix preservation in `die()` are concrete enough that an implementing agent can't accidentally modernize.
- **Circular-import hazard identified and resolved** with deferred function-body imports in both directions (detect ↔ parallel). The plan explicitly verifies this by importing both modules in either order.
- **Byte-identical parity gates** (Tasks 3 in both 01-02 and 01-03) are the right correctness proof for a mechanical migration, with distinct `scratch/` filenames that prevent cross-plan collision.
- **Test tier isolation is clean.** `tests/unit/` vs `tests/subprocess/` with no shared `conftest.py` avoids merge conflicts between parallel executors — a subtle but practical detail.
- **Env-const layering problem** (ICQ/QPMAX/DV_PROFILE) is correctly identified and the `monkeypatch.setattr(module, ...)` vs `monkeypatch.setenv` rule is documented. This is the kind of thing that silently breaks if missed.
- **Threat models are proportionate.** No over-engineering — `shell=True` is explicitly banned, the uv bootstrap risk is accepted with rationale, and the argv-injection surface is covered by the `fp` fixture tests.

## 3. Concerns

| Severity | Plan | Concern |
|----------|------|---------|
| **MEDIUM** | 01-02 | The parity script (Task 3) says "Use `--no-qsv`/software decode if QSV is unavailable" but `detect_scenes()` takes a `DetectionConfig`, not a CLI flag. The synthetic clip generated by `testsrc` has no QSV-specific codec, so `use_qsv=False` would work — but the script needs to construct `DetectionConfig(use_qsv=False)` explicitly. The current wording implies a CLI flag that doesn't exist in the library API. |
| **MEDIUM** | 01-03 | The encode parity script (Task 3) **requires real Arc hardware** (`qsvencc`, `/dev/dri`) to produce the oracle. The objective says "fast test tier" but this particular gate is hardware-gated by necessity — it should be explicitly marked as such (e.g., "hardware-OK, not a fast test") so CI can skip it when `/dev/dri` is absent. |
| **MEDIUM** | 01-03 | The encode parity script compares "encoded AV1 video streams byte-identically" but doesn't specify how to extract them from `.mkv` containers. `mkvmerge` muxing adds mux-level metadata (timestamps, track headers) that may differ between runs even with identical video payloads. The plan should compare the raw `.obu` files *before* muxing, or extract via `ffmpeg -c copy -bsf:v extract_extradata` — the exact extraction command should be specified. |
| **LOW** | 01-01 | `uv_build>=0.11.28,<0.12` is the build-backend constraint, but the plan doesn't account for the possibility that `uv lock` may resolve a *different* version than what's available at `uv sync` time on a different platform. `uv.lock` is cross-platform by default, but `uv_build` is a build dep — the lockfile should be generated and committed, then the verification should use `uv sync --locked` (which it does), so this is mostly a concern for the `uv lock` step itself. Minor. |
| **LOW** | 01-01 | The plan says "Do NOT add `[tool.ruff]`" per D-10/RESEARCH Open Question 3, but the RESEARCH document itself notes this is "allowed but optional." Not a concern per se, but the implementing agent should not interpret this as permanently forbidden — just deferred. |
| **LOW** | 01-02 | The detection migration routes `keyframes_in_window` subprocess calls through `proc.run`, but `keyframes_in_window` itself is called from `ProcessPoolExecutor` worker functions (`_boundary_worker`). If `proc.run` sets any global state (it currently doesn't), this would be a pickle-safety issue. Worth a one-line note in the action confirming that `proc.run` is stateless. |
| **LOW** | 01-03 | Seven test files in a single Task 2 is a large atomic unit. If one test file has a flaky assertion, the whole task blocks. Consider whether the plan should split this into sub-tasks (e.g., pure tests first, then mocked) — though the plan already marks it TDD, so this is more of a workflow suggestion than a correctness gap. |

## 4. Suggestions

1. **Encode parity — compare `.obu` before mux.** In Plan 01-03 Task 3, both legacy and migrated runs should be modified to output `movie.obu` files to a temp directory *before* the `mkvmerge` step. Compare those `.obu` files with `cmp`. Frame-count verification via `ffprobe` on the final `.mkv` is still valuable as a secondary check, but the primary byte-identical gate should be on the raw AV1 stream to eliminate mux metadata variance.

2. **Detection parity — make `use_qsv` explicit.** In Plan 01-02 Task 3, the parity script should construct `DetectionConfig(use_qsv=<probe_result>)` explicitly based on whether `/dev/dri` is present, rather than implying a `--no-qsv` CLI flag exists. Add a one-line check: `use_qsv = Path("/dev/dri/renderD128").exists()`.

3. **Add a `scratch/.gitignore` entry.** Both plans write synthetic media to `scratch/`. The plan should note that `scratch/` needs a `.gitignore` (or at minimum `*.mkv`, `*.scenes`, `*.obu` patterns) to prevent accidental commits of generated artifacts.

4. **Clarify `uv sync --locked` vs `pip install -e .` equivalence.** Plan 01-01's success criteria reference `pip install -e .` but the implementation uses `uv sync --locked`. These are equivalent *only* if the `uv` venv is activated. The verification command uses `uv run python -c ...` which handles this — good — but the acceptance criteria text should reference `uv sync --locked` (or `uv run pytest`) consistently rather than mixing `pip` and `uv` invocations.

5. **Document the `shared/logging.py` import path in Plan 01-01.** The plan says `from enpipe.shared.logging import die`, but the module is named `logging.py` which shadows stdlib `logging`. In migrated code, `from enpipe.shared import logging` works fine, but any code that does `import logging` (stdlib) followed by `from enpipe.shared.logging import die` would silently shadow. Recommend a brief note that migrated code must use the full qualified import, not a bare `import logging`.

## 5. Risk Assessment: **LOW**

The plans are tightly scoped, faithfully implement all 15 locked decisions, and have strong verification gates. The primary risk — silent behavior change during migration — is mitigated by the byte-identical parity scripts and the zero-logic-change discipline. The medium-severity concerns are all fixable with one-line clarifications before execution; none require re-planning. The wave dependency chain (01-01 → 01-02 → 01-03) is correct and the plans are ready for implementation.

## Plan 01-01: Package Scaffold + Shared Proc/Logging Seam

### Summary
Well-bounded foundation plan that correctly establishes the `uv`/`uv_build` lockfile toolchain, package skeleton, and the two shared leaf modules (`proc.py`, `logging.py`) that downstream waves depend on. Acceptance criteria are concrete and testable, with good decision traceability. The main risks are unverified dependency version pins and minor specification gaps around the `uv_build` configuration.

### Strengths
- Excellent decision traceability (D-01 through D-15 explicitly mapped)
- Correctly defers `[project.scripts]`, ruff/pyright, and CLI entry to later phases — no scope creep
- Verbatim line references to legacy source for the `proc.py`/`logging.py` seeds
- Acceptance criteria include the Pitfall 5 tripwire (`.venv` resolution, not system site-packages)
- Threat model is proportionate; T-01-01 (shell=True prohibition) is the right guard
- `die` prefix preservation (`encode_scenes: `) is explicitly flagged as parity surface — good awareness

### Concerns
- **MEDIUM — Unverified version pins.** `scenedetect==0.7`, `numpy==2.5.1`, `pytest==9.1.1`, `pytest-subprocess==1.6.0`, `pytest-mock==3.15.1`, `uv_build>=0.11.28,<0.12` are all stated as exact pins but the plan doesn't include a verification step that these versions exist on PyPI before `uv lock`. If any version doesn't exist, `uv lock` fails and the entire wave blocks. The plan should run `uv lock` early and fail fast with a clear diagnostic.
- **MEDIUM — `scenedetect==0.7` pin semantics.** PEP 440 zero-padding means `==0.7` matches `0.7.0` but NOT `0.7.1`. If the intent is "0.7.x latest" this is wrong; if the intent is "exactly 0.7.0" this is correct but should be written `==0.7.0` for clarity. D-02 says "whatever currently works" — verify which 0.7.x is actually installed before pinning.
- **MEDIUM — No `.gitignore` updates.** The plan creates `src/enpipe/` (new package tree) and relies on `.venv/` but doesn't update `.gitignore` for `__pycache__/`, `*.egg-info/`, `.ruff_cache/`, or build artifacts from `uv_build`. The existing `.gitignore` may not cover these.
- **LOW — Empty `__init__.py` files vs. convention.** CONVENTIONS.md says "Every module starts with `from __future__ import annotations`" and "Module docstrings are substantial." Empty `__init__.py` files for `detection/`/`encoding/`/`shared/` are fine for now, but clarify whether package `__init__.py` files are exempt from the convention or should carry a Russian docstring.
- **LOW — `requires-python >=3.12` allows 3.13+.** Legacy targets 3.12 specifically (devcontainer pins `python:3-3.12-trixie`). `>=3.12` is reasonable but if 3.13 introduces breaking changes in `scenedetect`/`numpy`, the lockfile won't guard against it. Consider `>=3.12,<3.13` or document that the devcontainer pins 3.12.

### Suggestions
- Add a pre-step in Task 1: run `uv lock --dry-run` or `pip index versions scenedetect` to verify all pinned versions exist before committing to them.
- Explicitly list `.gitignore` additions as a modified file or inline action.
- Specify the `[tool.uv]` section if `uv_build` requires package discovery config (e.g., `package-dir`, `packages`). The plan says "verbatim from RESEARCH" but the reviewer can't verify RESEARCH content.

### Risk Assessment: **LOW**
This is a scaffold plan with no existing behavior to break. The main risk is a version pin not existing on PyPI, which fails fast and is trivially fixable. No correctness invariants are at stake.

## Plan 01-02: Detection Migration + Tests + Parity

### Summary
Solid mechanical-migration plan with a clear line-level source→module map, explicit circular-import resolution, and a byte-identical parity gate against the legacy oracle. The test tier covers the right pure-logic and subprocess-boundary targets per D-11. Main risks are the non-mechanical nature of the deferred-import circular-import fix, weak synthetic-clip scene diversity, and missing `.gitignore` for `scratch/`.

### Strengths
- Line-level source→module map is precise and copy-pasteable — minimizes migration drift
- Circular import (detect↔parallel) is explicitly addressed with Pattern 2 deferred imports in both directions
- Test fixtures kept inline to avoid `conftest.py` conflicts with Plan 01-03 — good cross-plan coordination
- Distinct `scratch/parity_detect_sample.mkv` filename to avoid collision with 01-03
- `probe_source` error-path test asserts `SceneDetectionError` not `SystemExit` — correctly validates the typed-exception regime
- TDD flag on Task 2; acceptance criteria assert exact argv (not substring) — good test rigor

### Concerns
- **MEDIUM — Deferred imports are not purely mechanical.** The legacy code is a single file with no circular import. Introducing `from .parallel import detect_scenes_parallel` inside `detect_scenes` (and vice versa) is a structural change to make the module split work. This is necessary and minimal, but it's in tension with D-13's "ZERO logic changes." The plan should explicitly acknowledge this as a sanctioned deviation rather than implying the migration is 100% mechanical.
- **MEDIUM — Synthetic clip may not produce scene cuts.** `testsrc=duration=10:size=640x360:rate=24` generates a gradual test pattern. PySceneDetect's `AdaptiveDetector` may not fire any cuts, producing a trivial single-scene `.scenes` file. The parity check would still be valid (byte-identical trivial output), but it wouldn't exercise the scene-boundary splitting logic. Consider `testsrc2` or concatenating distinct `lavfi` sources (e.g., `color` → `smptebars` → `mandelbrot`) to force detectable cuts.
- **MEDIUM — No `.gitignore` for `scratch/`.** The parity script and synthetic clip land in `scratch/` with no mention of gitignoring the directory. If committed, the binary `.mkv` pollutes the repo. Add `scratch/` to `.gitignore` (or specify which files are kept vs. gitignored).
- **LOW — QSV availability handling in parity script.** The script calls `python3 legacy/scene_detection.py scratch/parity_detect_sample.mkv` without `--no-qsv`. The plan says "hardware is present" but doesn't handle the case where QSV init fails (driver issue, `/dev/dri` permissions). Add a fallback: try with QSV, retry with `--no-qsv` on failure, or detect `vainfo` first.
- **LOW — Thin test coverage.** Only `_min_scene_len`, `_build_scenes` (TEST-01) and `probe_source` (TEST-02) are tested. `_detect_relative`, `keyframes_in_window`, `find_boundary`, and `_sanitize_boundaries` have no tests. D-12 says "don't chase main() coverage" but these are pure-logic functions, not CLI glue. Consider adding at least `_sanitize_boundaries` (boundary stitching is correctness-critical for parallel detection).

### Suggestions
- Use a multi-source concatenated synthetic clip to force scene cuts: `ffmpeg -f lavfi -i color=red:duration=3:size=640x360:rate=24 -f lavfi -i color=blue:duration=3:size=640x360:rate=24 -f lavfi -i smptebars=duration=4:size=640x360:rate=24 -filter_complex "[0][1][2]concat=n=3" ...`
- Add a `--no-qsv` retry path in the parity script for robustness.
- Add `scratch/` to `.gitignore` as an explicit action item in Task 3.

### Risk Assessment: **LOW-MEDIUM**
Detection migration is lower-risk than encoding because the output is text (`.scenes`), not binary encode output. The circular-import fix is well-understood. The main risk is a trivial parity check that doesn't exercise scene-cut boundaries, giving false confidence in the migration.

## Plan 01-03: Encoding Migration + Tests + Parity

### Summary
The most complex and highest-risk plan: 7 modules + 7 test files + a real-hardware encode parity gate. The line-level migration map is thorough, env-const preservation and worker-thread regime are explicitly called out, and EBML is correctly kept inline (D-07). However, two significant concerns threaten the plan's viability: (1) the byte-identical encode parity assumption depends on qsvencc determinism, which is unverified and may not hold for hardware AV1 encoders, and (2) stripping the preflight block from `run_encode` is an undocumented behavior change in a "zero behavior change" phase.

### Strengths
- Comprehensive 7-module line-level source→module map with exact legacy line ranges
- `import re` moved to top of `scenes_io.py` — correctly fixes legacy's mid-file import per CONVENTIONS
- EBML parser explicitly kept inline (D-07) — no premature extraction
- Env-const preservation called out with specific tripwire values (`ICQ==23`, `DV_PROFILE=="10.1"`) — Pitfall 3 guard
- Worker-thread tuple regime (`encode_audio`/`encode_chunk` never call `die()`) explicitly preserved
- `run_encode(args)` shape with legacy attribute names is Phase-4-ready without overbuilding
- Threat model covers the env-var ↔ encode-parameter boundary (T-01-E2)
- 14-file count explicitly justified as inherent to whole-stage migration

### Concerns
- **HIGH — Byte-identical encode parity assumes qsvencc determinism.** The plan asserts `cmp` byte-identical on the encoded video stream between legacy and migrated runs. If the migration is truly mechanical (same qsvencc argv), parity holds **only if qsvencc is deterministic** — i.e., same input + same argv → identical bytes. Hardware AV1 encoders are often non-deterministic due to thread scheduling, GPU state, and rate-control feedback loops. If qsvencc isn't deterministic, the parity check fails spuriously even with a perfect migration, blocking the phase. The plan must either: (a) verify determinism first (run legacy twice, `cmp` the outputs), or (b) provide a fallback parity criterion (frame counts + SSIM/PSNR metrics match within tolerance) when byte-identical fails.
- **HIGH — Preflight stripping is an undocumented behavior change.** The plan says "strip the `argparse`/`parse_args`/preflight block" from `main()` → `run_encode()`. The preflight (`shutil.which` tool-availability check) is a behavior: it `die()`s early with a clear message if a tool is missing. Removing it means `run_encode` fails later with a confusing `FileNotFoundError` or subprocess error. This contradicts D-13's "ZERO logic changes." The plan should either: (a) keep preflight in `run_encode`, or (b) explicitly document that preflight moves to the Phase 4 CLI wrapper and is intentionally excluded from the library function (with the parity script replicating it externally).
- **MEDIUM — No determinism pre-check for autonomous execution.** The plan is `autonomous: true`. If the byte-identical parity check fails due to qsvencc non-determinism, the autonomous executor has no fallback and the phase blocks. Add a determinism pre-check (run legacy twice, `cmp`) as a prerequisite to the parity comparison, with a documented fallback criterion.
- **MEDIUM — All 7 modules migrated in one task.** Task 1 migrates all 7 encoding modules before any test runs. A single import error or migration mistake in one module blocks all 7 tests. Consider incremental migration within Task 1: migrate `scenes_io` → import check; migrate `keyframes` → import check; etc. This localizes failures.
- **MEDIUM — HDR/DV parity untested.** The synthetic clip (`testsrc`) is SDR. The encode parity check doesn't exercise HDR10/HDR10+/Dolby Vision detection or `--dolby-vision-rpu copy`. The mocked `detect_hdr` TEST-02 test covers the logic, but there's no end-to-end HDR parity. This is acceptable for Phase 1 (real HDR media testing is Phase 4/TEST-04) but should be explicitly noted as a parity gap.
- **MEDIUM — Video stream extraction method unspecified.** The parity check says "extract each `movie.obu`/video track and `cmp`" but doesn't specify the extraction command (`mkvextract`? `ffmpeg -c copy -f av1`? `qsvencc` raw output?). The extraction method matters because mkvmerge may reorder/mux differently. The plan should specify comparing the raw `.obu` concatenation (before mkvmerge) if possible, since that's what the high-water-mark append produces.
- **LOW — `chunk_command` purity vs. env-const reads.** `chunk_command` is described as a "pure argv builder" tested without `fp` fixture. But it reads module-level constants (`ICQ`, `QPMAX`, `GOP_LEN`) which are set from env vars at import time. The test needs `monkeypatch.setattr` on these (Pattern 4) — the plan mentions Pattern 4 generically but doesn't explicitly flag that `chunk_command` tests need it.
- **LOW — `_START` timing shift.** `_START = time.monotonic()` in `shared/logging.py` is set at import time, which may differ from legacy's import time. This doesn't affect parity (log output isn't compared) but changes elapsed-time prefixes in log messages. Note as a known cosmetic difference.

### Suggestions
- **Critical: Add a qsvencc determinism pre-check.** Before the parity comparison, run `legacy/encode_scenes.py` twice on the same input and `cmp` the outputs. If they differ, fall back to frame-count + SSIM/PSNR metrics parity (within 0.0 tolerance for frame counts, within epsilon for SSIM). Document this fallback explicitly.
- **Keep preflight in `run_encode` or document its exclusion.** Either preserve the `shutil.which` checks as the first thing `run_encode` does (simplest, most parity-preserving), or add a note: "Preflight moves to the Phase 4 CLI wrapper; `run_encode` assumes tools are available. The parity script calls preflight externally before `run_encode`."
- **Compare raw `.obu` before mkvmerge, not the final `.mkv`.** The high-water-mark append produces `movie.obu` — compare that directly (it's pre-mux, so mkvmerge packaging differences don't interfere). Then separately verify frame counts on the final `.mkv`.
- **Migrate modules incrementally with import checks** between each to localize failures.
- **Note HDR parity as a known gap** with a forward reference to Phase 4/TEST-04.

### Risk Assessment: **MEDIUM-HIGH**
The encoding migration is the phase's riskiest plan. The qsvencc determinism issue is the single biggest threat — it could block phase completion autonomously with no fallback. The preflight stripping is a correctness-adjacent behavior change that should be documented or reversed. The 7-module-at-once migration increases blast radius for migration errors. These are addressable with the suggested fixes, but as written, the plan has a non-trivial probability of failing its own acceptance criteria spuriously.

## Cross-Cutting Observations

### Scope Discipline
All three plans maintain excellent scope discipline. EBML isolation, seek/trim extraction, ThreadPool/ProcessPool resolution, the regression test, CI, and the unified CLI are correctly deferred to later phases. No scope creep detected.

### Wave Ordering
The wave dependency chain (01-01 → 01-02 → 01-03) is correct: the shared seam must exist before migration, and detection migrates before encoding (smaller, no dependents). The `depends_on` fields are accurate.

### Parity Verification Strategy
The parity approach is sound in principle but has a fundamental weakness: it assumes deterministic output from external tools. For detection (text output), this is safe. For encoding (binary hardware encode output), this is risky. **The phase should establish a determinism baseline before relying on byte-identical parity.**

### Test Adequacy
The fast test tier (TEST-01 + TEST-02) covers the right targets but is thin on detection-side pure-logic coverage (`_sanitize_boundaries`, `find_boundary` untested). The encoding-side coverage is more complete. The tests will catch argv drift and obvious logic errors but won't catch subtle migration mistakes in untested functions — the parity gate is the backstop for those.

### Missing Across All Plans
- **No `.gitignore` updates** for `scratch/`, `__pycache__/`, `*.egg-info/`, build artifacts
- **No determinism baseline** for external tool output before relying on byte-identical parity
- **No explicit acknowledgment** that deferred imports (01-02) and preflight stripping (01-03) are sanctioned deviations from "ZERO logic changes" (D-13)

### Overall Phase Risk: **MEDIUM**
The plans are well-structured and traceable, with good scope discipline. The encoding parity gate (01-03 Task 3) is the critical path risk: if qsvencc is non-deterministic, the phase blocks with no fallback. Adding a determinism pre-check and a metrics-based fallback would reduce this to LOW-MEDIUM. The preflight stripping in 01-03 should be documented or reversed to maintain the "zero behavior change" contract.
