<objective>
Capture the mandatory parallel==sequential regression baseline (TEST-03) against the DEBT-03-resolved implementation, and make it un-fakeable — the test must PROVE the parallel branch ran, not just that some output matched.

Purpose: This test is the guard that DEBT-03's executor resolution did not change detection output, and the standing guard against any future regression in the multi-segment stitching path (`_sanitize_boundaries`, `non_cut_offsets` merge). Per D-05/D-06 it uses the software-decode fallback so it runs in ordinary CI without GPU. `detect_scenes_parallel` silently falls back to sequential in TWO places (verified in code): (1) the gate at parallel.py:121 (`total < jobs * min_span`, min_span = max(2*min_scene_len, round(60*fps)) = 1440 frames = 60s at 24fps) AND (2) the collapse at parallel.py:133 (`if len(bnds) < 3`). A too-short clip trips (1); a broken `find_boundary` trips (2) — and BOTH fallbacks also yield >=2 scenes, so a `>=2 scenes` guard cannot tell a real parallel run from a silent fallback (opencode HIGH). This plan therefore adds: a pure gate-arithmetic unit test; an executor-agnostic engagement assertion in the regression test (the deferred sequential fallback `detect_scenes` was NEVER invoked during the parallel run — required and unconditional, valid under BOTH DEBT-03 outcomes since both fallbacks route through it in the PARENT process before any executor/pickling), optionally refined by a ThreadPool-only `_segment_worker` call-count check (SKIPPED under ProcessPoolExecutor, where `mocker.spy` on a module-level worker autospecs it and the autospec'd mock cannot be pickled through `ex.map`, hard-crashing the call with PicklingError); a focused pure unit test for the `non_cut_offsets` merge; and jobs=[2,3] parameterization so multi-boundary stitching is exercised.

Output: Three committed tests in the default (non-hardware) tier.
</objective>

Executor-dependency note (from Plan 03-01's decision) — REQUIRED conditioning:
`detect_scenes_parallel` reaches its sequential fallback via a DEFERRED `from .detect import detect_scenes`
inside the function body (parallel.py:122 and :134), so the fallback name resolves against the
`enpipe.detection.detect` module at call time — spy `enpipe.detection.detect.detect_scenes` (NOT a
parallel-module attribute; `detect_scenes` is NOT imported into the parallel namespace). This fallback ALWAYS
runs in the PARENT process before any executor is created and is never pickled, so "detect_scenes NOT invoked
during the parallel run" is a valid engagement proof under BOTH ThreadPoolExecutor and ProcessPoolExecutor.
Make THIS the REQUIRED, UNCONDITIONAL engagement assertion.
The `_segment_worker` call-count check is only valid under an IN-PROCESS executor. `_segment_worker` is
module-level and dispatched via `ex.map` (parallel.py:147). If 03-01 SWITCHED to ProcessPoolExecutor,
`mocker.spy(parallel, "_segment_worker")` autospecs the module-level worker, and an autospec'd mock CANNOT be
pickled through `ProcessPoolExecutor.map` — this HARD-CRASHES `detect_scenes_parallel(jobs=N)` with
PicklingError (verified empirically) for EVERY parameterized jobs value, not merely under-counts, breaking
TEST-03 and CI-01. Therefore the `_segment_worker` spy MUST be gated: inspect which Executor class
`enpipe.detection.parallel` binds after 03-01 (e.g. whether it binds/uses `ProcessPoolExecutor`) and perform
the `mocker.spy(parallel, "_segment_worker")` + `call_count > 1` assertion ONLY when the active executor is a
`ThreadPoolExecutor`; SKIP it under ProcessPool (the required detect_scenes no-fallback proof already covers
engagement there).
